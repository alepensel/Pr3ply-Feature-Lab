# Security and Privacy

## Public-readiness checklist

- [x] `.env` is absent from the sanitized repository history. `.env.example` contains variable names with blank values only.
- [x] No literal project identifiers, API keys, tokens, or server secrets are committed. Browser configuration is injected by the hosting environment.
- [x] All server secrets (Agora App Certificate, Google OAuth client secret, Lovable AI key, service-role key) live in Lovable Cloud edge-function secrets and are never referenced by literal value in source.
- [x] Every public-schema table has RLS enabled and explicit `GRANT`s.
- [x] Roles are stored in a dedicated `user_roles` table; client write grants are revoked; `has_role` is self-scoped.
- [x] Realtime is allow-listed to `room-state-%` topics and requires session participation.
- [x] Agora joins require a token issued by the `agora-token` edge function after tutor/booking verification.
- [x] Google Calendar OAuth `state` is HMAC-signed; callback validates `redirectTo` is a relative path.
- [x] Booking capacity is enforced by a `BEFORE INSERT` trigger with a row lock (not just by the UI).
- [x] Edge functions return generic error strings to the client; specifics stay in server logs.
- [x] `TutorDashboard` performs a client-side role guard in addition to server-side RLS.

## Privacy considerations

- **Profile PII:** name, avatar, birth country, current country. Participant display fields are available only to the session tutor and learners with confirmed bookings (see decision D9).
- **Transcripts:** text only, generated in-browser via the Web Speech API. Stored in `session_transcripts` and readable only by the owning learner and the session tutor.
- **Audio:** the application does not upload or store audio. Depending on the browser, speech recognition may be processed by a browser or platform service.
- **Feedback reports:** readable only by the owning learner and the session tutor.
- **Google Calendar:** per-user OAuth; tokens are scoped to the connecting user's row in `google_calendar_connections`; disconnect deletes them.

## AI governance

- **Inputs sent to the model** for prompts: session `theme`, `scenario`, `description`, `level`, `participant_count`. No user PII.
- **Inputs sent to the model** for feedback: the learner's own transcript, session context, and the prompts used. Sent per-learner to isolate leakage risk.
- **Structured output** is enforced via tool-calling; the client renders known fields only.
- **Fallback:** if AI generation fails, the UI degrades to a manual retry state; the session is never blocked.
- **Future work:** learner-facing transparency ("what data is used to generate my feedback"), retention policy for transcripts (currently retained indefinitely), and an opt-out.

## Summary of resolved scanner findings

All previously reported findings have been fixed. The main ones:

- Agora tokens now issued server-side (was `null`).
- Overbooking prevented at the database (was UI-only).
- `has_role` restricted to self-checks (was arbitrary lookup).
- `user_roles` writes revoked from client roles (prevented self-escalation).
- Google Calendar OAuth `state` HMAC-signed (prevented UID spoofing) and `redirectTo` validated (prevented open redirect).
- Realtime restricted to `room-state-%` with participation check (prevented cross-session subscription).
- Edge functions return opaque error messages (prevented info leakage).
- `TutorDashboard` role guard added on the client (defense-in-depth over RLS).
